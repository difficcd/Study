# 기본 지식 정리
## RAG, sLLM
###  1. RAG부터 — 오해 하나 바로잡기

RAG(Retrieval-Augmented Generation, 검색증강생성)는 **MCP랑은 다른 거예요.** 여기서부터 정리해야 나머지가 풀립니다.

**RAG의 핵심 아이디어:** LLM이 답을 생성하기 전에, **외부 문서에서 관련 내용을 먼저 검색해서, 그걸 프롬프트에 끼워 넣고 답하게 하는 것.**

왜 필요하냐면, LLM은 학습 시점까지의 지식만 갖고 있고(앞서 본 knowledge cutoff), 회사 내부 문서나 최신 정보는 모르잖아요. 그걸 매번 검색해서 "이 자료 참고해서 답해"라고 넣어주는 거예요.

흐름:

```
사용자 질문
   ↓
① 검색(Retrieval): 질문과 관련된 문서 조각을 DB에서 찾음
   ↓
② 증강(Augmented): 찾은 문서를 프롬프트에 붙임
   "다음 자료를 참고해 답해: [문서내용] / 질문: ..."
   ↓
③ 생성(Generation): LLM이 그 자료 기반으로 답 생성
```

**MCP와의 차이:** MCP는 Claude 같은 모델이 외부 도구(검색, 파일, API)에 **연결되는 통로/프로토콜**이에요. "어떻게 연결하느냐"의 규격이죠. RAG는 "검색해서 그 결과로 답을 보강한다"는 **방법론/아키텍처**고요.

- RAG = **무엇을 하는가** (검색→증강→생성이라는 전략)
- MCP = **어떻게 연결하는가** (도구 연결 규격)

MCP를 써서 RAG를 구현할 수도 있지만, MCP 없이도 RAG는 얼마든지 만듭니다(직접 검색 코드 짜서). 그러니 "RAG = MCP로 검색 잘하게 하는 것"은 아니에요. RAG가 더 큰 개념이고 더 오래됐어요(MCP는 2024년 등장, RAG는 그 전부터 있었음).

전통적 RAG는 보통 **임베딩 기반 벡터 검색**을 써요. 문서를 잘게 쪼개 임베딩 벡터(앞서 배운 그 임베딩!)로 바꿔 DB에 저장해두고, 질문도 임베딩해서 **벡터 유사도가 높은 문서 조각**을 찾아오는 방식입니다.


### 2. sLLM이 뭐냐

**sLLM = small Large Language Model** (작은 거대언어모델). 말이 모순 같지만, **상대적으로 크기가 작은 LLM**을 뜻해요.

- **큰 LLM**: GPT-4, Claude 같은 수천억 파라미터급. 성능 최고지만 **직접 못 돌리고 API로만** 씀, 비용 듦.
- **sLLM**: LLaMA, Mistral, Gemma 같은 **공개된**(open) 비교적 작은 모델(수B~수십B). **내 서버/PC에 직접 띄워서** 돌릴 수 있어요.

sLLM의 매력은 **내가 소유·통제**한다는 거예요. 데이터를 외부(OpenAI 등)에 안 보내도 되고, 비용도 통제되고, **내 입맛대로 파인튜닝**할 수 있어요. 대신 성능은 거대 모델보다 떨어질 수 있죠.


### 3. "LLM을 API로 사용해 RAG 구현"이 뭔 소리

RAG를 만들 때 **③ 생성** 단계에서 어떤 LLM을 쓸지 두 갈래가 있어요:

**방법 A — API 방식 (ChatGPT API 등)**

- 검색은 내가 하고, 생성만 **OpenAI 서버에 요청**해서 받아옴.
- `검색된 문서 + 질문`을 OpenAI API로 보내면 → GPT-4가 답을 만들어 돌려줌.
- **장점**: 모델이 이미 똑똑해서 **파인튜닝 안 해도** 검색 자료만 잘 넣어주면 답을 잘 함.
- **단점**: 비용, 데이터 외부 전송, 커스터마이징 한계.

**방법 B — sLLM 직접 방식**

- 생성도 **내 서버의 공개 LLM**(LLaMA 등)으로 함.
- **장점**: 소유·통제·보안.
- **단점**: 작은 모델이라 **그냥 쓰면 검색 자료를 잘 못 활용**할 수 있음. 그래서 **"이 모델이 검색 결과를 잘 읽고 답하도록" 파인튜닝(추가 학습)**이 필요할 수 있어요.

### 4. 이제 그 문장 해석

> "ChatGPT API 같은 고성능 LLM을 API로 써서 RAG를 구현하는 게 아니라, sLLM 같은 공개 LLM으로 RAG를 구현하려면, sLLM을 RAG를 위해 파인튜닝하는 방법을 고민하는 게 더 중요하다."

풀어쓰면:

- **GPT-4 API로 RAG 만들면** → 모델이 워낙 똑똑해서 파인튜닝 고민 별로 안 해도 됨. 검색만 잘 붙이면 알아서 잘 답함.
- **하지만 공개 sLLM으로 RAG 만들려면** → 모델이 상대적으로 약하니까, **"검색된 문서를 잘 참고해서 답하는 능력"을 따로 학습(파인튜닝)시켜야** 제 성능이 나옴. 그래서 이 경우엔 _파인튜닝 방법_이 더 중요한 고민거리가 된다.

즉 **"어떤 LLM을 쓰느냐에 따라 RAG 구축 시 신경 쓸 포인트가 다르다"**는 얘기예요. 강력한 API 모델이면 검색 품질에 집중, 약한 공개 모델이면 모델 파인튜닝에도 신경 써야 한다는 것.

### 정리

|용어|뜻|
|---|---|
|**RAG**|검색→문서를 프롬프트에 증강→생성. LLM에 외부지식 붙이는 전략|
|**MCP**|모델을 외부 도구에 연결하는 규격. RAG와 별개(수단 중 하나일 뿐)|
|**sLLM**|직접 돌릴 수 있는 작고 공개된 LLM (LLaMA 등)|
|**API로 RAG**|생성을 GPT-4 등 외부 API에 맡김. 똑똑해서 파인튜닝 부담 적음|
|**sLLM으로 RAG**|생성을 내 공개모델로. 약해서 파인튜닝이 더 중요해짐|

핵심 교정: **RAG ≠ MCP로 검색 잘하기.** RAG는 "검색 결과를 프롬프트에 넣어 생성을 보강하는 더 큰 전략"이고, 그 안에서 검색 방식·생성 모델 선택이 갈리는 거예요.


## LLM 튜닝 비용? 도구?

## LLM 공부 순서는?
기본적으로는 인공지능에서 배운 순서 그대로 배워도 좋을 것 같지만 그보다 더 상세한 순서이니 참고.

> LLM이라고 하는 큰 주제만 본다면, 이론적으로 크게는 트랜스포머(Transformer), BERT, GPT, BART, T5, GPT-3, Parameter-Efficient Fine-Tuning (PEFT), LLama 등이 큰 그림에서 공부할 수 있는 키워드라고 생각합니다. 실무적으로는 BERT, T5의 태스크 별 파인 튜닝 이후 LLM 파인 튜닝을 해보시면 그래도 전체적인 LM에 대한 흐름을 이해할 수 있을 것 같습니다.



## sLLM은 어디까지 사용 가능한가?

> 실제로 저자 및 주변에서 여러 서비스를 내고 있으며, SFT 튜닝 또는 LoRA 튜닝 후에 특정 태스크 한정으로 이미 사례들이 존재합니다. 번역, 요약, 페르소나 챗봇, text-to-sql, 심지어 RAG용으로도 (잘 정제된 데이터셋으로 **RAFT (Retrieval Augmented Fine-tuning)** 튜닝을 통해 할루시네이션을 극도로 축소하여) 가능합니다. 사실 10B 이내의 sLLM을 튜닝없이 구동하여 써보고 GPT-4에 비해 성능이 떨어지니 서비스에는 쓸 수 없다고 하는 말은 (적어도 NLP 엔지니어로서는) 맞지 않는 행동이라고 생각합니다. 튜닝 전과 후는 체감 성능이 최대 배 이상 차이나는 다른 모델일 수 있습니다.
