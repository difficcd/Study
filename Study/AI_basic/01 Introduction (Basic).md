# 01 ML Workflow

https://wikidocs.net/217160
데이터 사이언스(Data Science) 또는 머신 러닝(Machine Learning) 과정에서 거치는 전반적인 과정 ( 딥 러닝 또한 머신 러닝의 한 갈래로 **딥 러닝 워크플로우** 또한 **머신 러닝 워크플로우**)

![[Pasted image 20260630162317.png|483]]

### 1) 수집(Acquisition)

머신 러닝을 하기 위해서는 기계에 학습시켜야 할 데이터가 필요합니다. 자연어 처리의 경우, 자연어 데이터를 **말뭉치 또는 코퍼스(corpus)** 라고 부르는데 코퍼스의 의미를 풀이하면, << 조사나 연구 목적에 의해서 특정 도메인으로부터 수집된 텍스트 집합 >>을 말합니다. 텍스트 데이터의 파일 형식은 txt 파일, csv 파일, xml 파일 등 다양하며 그 출처도 음성 데이터, 웹 수집기를 통해 수집된 데이터, 영화 리뷰 등 다양합니다.

### 2) 점검 및 탐색(Inspection and exploration)

데이터가 수집되었다면, 이제 데이터를 점검하고 탐색하는 단계입니다. 여기서는 데이터의 구조, 노이즈 데이터, 머신 러닝 적용을 위해서 데이터를 어떻게 <<정제>>해야하는지 등을 파악해야 합니다.

이 단계를 **<< 탐색적 데이터 분석(Exploratory Data Analysis, EDA) 단계 >>** 라고도 하는데 이는 독립 변수, 종속 변수, 변수 유형, 변수의 데이터 타입 등을 점검하며 데이터의 특징과 내재하는 구조적 관계를 알아내는 과정을 의미합니다. 이 과정에서 시각화와 간단한 통계 테스트를 진행하기도 합니다.


### 3) 전처리 및 정제(Preprocessing and Cleaning)

데이터에 대한 파악이 끝났다면, 머신 러닝 워크플로우에서 가장 까다로운 작업 중 하나인 데이터 전처리 과정에 들어갑니다. 이 단계는 많은 단계를 포함하고 있는데, 가령 자연어 처리라면 << 토큰화, 정제, 정규화, 불용어 제거 등의 단계 >> 를 포함합니다. <<< 빠르고 정확한 데이터 전처리를 하기 위해서는 사용하고 있는 툴(이 책에서는 파이썬)에 대한 다양한 라이브러리에 대한 지식이 필요 >>>합니다. 정말 까다로운 전처리의 경우에는 <<전처리 과정에서 머신 러닝이 사용되기도>> 합니다.


### 4) 모델링 및 훈련(Modeling and Training)

데이터 전처리가 끝났다면, 머신 러닝에 대한 코드를 작성하는 단계인 모델링 단계에 들어갑니다. 적절한 머신 러닝 알고리즘을 선택하여 모델링이 끝났다면, 전처리가 완료 된 데이터를 머신 러닝 알고리즘을 통해 기계에게 학습(training)시킵니다. 이를 훈련이라고도 하는데, 이 두 용어를 혼용해서 사용합니다. 

기계가 데이터에 대한 학습을 마치고나서 훈련이 제대로 되었다면 그 후에 기계는 우리가 원하는 태스크(task)인 기계 번역, 음성 인식, 텍스트 분류 등의 자연어 처리 작업을 수행할 수 있게 됩니다.

여기서 주의해야 할 점은 대부분의 경우에서 모든 데이터를 기계에게 학습시켜서는 안 된다는 점입니다. 뒤의 실습에서 보게되겠지만 데이터 중 일부는 테스트용으로 남겨두고 훈련용 데이터만 훈련에 사용해야 합니다. 그래야만 기계가 학습을 하고나서, 테스트용 데이터를 통해서 현재 성능이 얼마나 되는지를 측정할 수 있으며 과적합(overfitting) 상황을 막을 수 있습니다. 사실 최선은 훈련용, 테스트용으로 두 가지만 나누는 것보다는 훈련용, 검증용, 테스트용. 데이터를 이렇게 세 가지로 나누고 훈련용 데이터만 훈련에 사용하는 것입니다.

=> 인공지능에서 배운 내용.

> 이 책에서는 실습 상황에 따라서 훈련용, 검증용, 테스트용 세 가지를 모두 사용하거나 때로는 훈련용, 테스트용 두 가지만 사용하기도 합니다. 하지만 현업에서 최선은 검증용 데이터 또한 사용하는 것임을 기억해둡시다.

- **3분할**: 훈련용(train) + **검증용(validation)** + 테스트용(test)

| 역할       |                                                                                       |
| -------- | ------------------------------------------------------------------------------------- |
| **훈련용**  | 모델 파라미터(가중치)를 직접 학습시키는 데 씀                                                            |
| **검증용**  | 학습 중간중간 **"지금 모델이 잘 되고 있나"를 체크**하고, 하이퍼파라미터(학습률, 레이어 수 등) 조정할 때 씀. **가중치 학습에는 안 들어감** |
| **테스트용** | 모든 게 끝난 후, **최종 성능을 딱 한 번** 확인. 절대 그 전까진 안 봄                                          |

### 5) 평가(Evaluation)

미리 언급하였는데, 기계가 다 학습이 되었다면 테스트용 데이터로 성능을 평가하게 됩니다. 평가 방법은 기계가 예측한 데이터가 테스트용 데이터의 실제 정답과 얼마나 가까운지를 측정합니다.


### 6) 배포(Deployment)

평가 단계에서 기계가 성공적으로 훈련이 된 것으로 판단된다면 완성된 모델이 배포되는 단계가 됩니다. 다만, 여기서 완성된 모델에 대한 전체적인 피드백으로 인해 모델을 업데이트 해야하는 상황이 온다면 수집 단계로 돌아갈 수 있습니다.

Pandas는 총 세 가지의 데이터 구조를 사용합니다.

- 데이터프레임(DataFrame)

이 중 데이터프레임이 가장 많이 사용되며 여기서는 시리즈와 데이터프레임에 대해서 다룹니다.

### 1) 시리즈(Series)

시리즈 클래스는 <<1차원>> 배열의 값(values)에 각 값에 대응되는 인덱스(index)를 부여할 수 있는 구조를 갖고 있습니다.

```python
import pandas as pd
sr = pd.Series([17000, 18000, 1000, 5000],
               index=["피자", "치킨", "콜라", "맥주"])

print('시리즈 출력 :')
print('-'*15)
print(sr)

print('시리즈의 값 : {}'.format(sr.values))
print('시리즈의 인덱스 : {}'.format(sr.index))
```

```python
시리즈 출력 :
---------------
피자    17000
치킨    18000
콜라     1000
맥주     5000
dtype: int64


시리즈의 값 : [17000 18000  1000  5000]
시리즈의 인덱스 : Index(['피자', '치킨', '콜라', '맥주'], dtype='object')
```


### 2) 데이터프레임(DataFrame)

데이터프레임은 <<2차원>> 리스트를 매개변수로 전달합니다. 2차원이므로 행방향 인덱스(index)와 열방향 인덱스(column)가 존재합니다. 다시 말해 행과 열을 가지는 자료구조입니다. 

시리즈가 인덱스(index)와 값(values)으로 구성된다면(기본 리스트/딕셔너리 느낌), 
데이터프레임은 열(columns)까지 추가되어 << 열(columns), 인덱스(index), 값(values)으로 구성됩니다 (DB 느낌? 3차원 느낌.) >>. 이 세 개의 구성 요소로부터 데이터프레임을 생성해봅시다.

```python
values = [[1, 2, 3], [4, 5, 6], [7, 8, 9]]
index = ['one', 'two', 'three']
columns = ['A', 'B', 'C']

df = pd.DataFrame(values, index=index, columns=columns)

print('데이터프레임 출력 :')
print('-'*18)
print(df, '\n\n')

# .format(df.요소:index,values,columns.) 를 통해 걸러냄
print('데이터프레임의 인덱스 : {}'.format(df.index))
print('데이터프레임의 열이름: {}'.format(df.columns))
print('데이터프레임의 값 :')
print('-'*18)
print(df.values)
```

```python
데이터프레임 출력 :
------------------
       A  B  C
one    1  2  3
two    4  5  6
three  7  8  9

데이터프레임의 인덱스 : Index(['one', 'two', 'three'],   
                           dtype='object')
데이터프레임의 열이름: Index(['A', 'B', 'C'], dtype='object')
데이터프레임의 값 :
------------------
[[1 2 3]
 [4 5 6]
 [7 8 9]]
```

=> "semantics를 가진 행렬"보다 한 단계 더 복잡한 구조라고 보는 게 맞음.
컬럼마다 데이터 타입이 다를 수 있음(heterogeneous). 또, 행과 열에 label=index 가 붙어 의미 있는 이름으로 네이밍되며 내부 구현이 사실 컬럼지향(열벡터 느낌 => 행렬보다는 열벡터를 묶어둔 것, 각 컬럼을 독립 1차원 arr로 취급한다고 생각하면 됨.) 
`df.values`, `df.to_numpy()`로 실제 행렬(ndarray)로 변환한다고 생각할 수도 있고
보통 이런 애들은 학습데이터를 텐서 변환하기 전처리 단계에서 사용됨 (데이터프레임을 거치며 데이터를 라벨 기반으로 정제)

### 3) 데이터프레임의 생성

데이터프레임은 리스트(List), 시리즈(Series), 딕셔너리(dict), Numpy의 ndarrays, 또 다른 데이터프레임으로부터 생성할 수 있습니다. 여기서는 리스트와 딕셔너리를 사용하여 데이터프레임을 생성해보겠습니다. 우선 이중 리스트로 생성하는 경우입니다.

```python
import pandas as pd

# 리스트로 생성하기
data = [
    ['1000', 'Steve', 90.72], 
    ['1001', 'James', 78.09], 
    ['1002', 'Doyeon', 98.43], 
    ['1003', 'Jane', 64.19], 
    ['1004', 'Pilwoong', 81.30],
    ['1005', 'Tony', 99.14],
]

df = pd.DataFrame(data)
print(df)
print('-'*25)

# 열이름 지정 가능
df = pd.DataFrame(data, columns=['학번', '이름', '점수'])
print(df)
```

### 4) 데이터프레임 조회하기

- df.head(n) - 앞 부분을 n개만 보기
- df.tail(n) - 뒷 부분을 n개만 보기
- df['열이름'] - 해당되는 열을 확인

위에서 사용한 데이터프레임을 그대로 사용하여 실습해봅시다.


```python

print(df.tail(3))

```

### 2) ndarray의 초기화

위에서는 리스트를 가지고 ndarray를 생성했지만 ndarray를 만드는 다양한 다른 방법이 존재합니다. 이 외에도 다양한 방법이 존재하므로 필요에 따라서 다양한 배열을 생성할 수 있습니다.

np.zeros()는 배열의 모든 원소에 0을 삽입합니다.
np.ones()는 1, np.full은 사용자가 넣는 값, np.eye는 단위행렬 생성함 (identity:I)
(단위행렬은 보통 초기화, 정규화, 원핫 인코딩에 사용함.)

**np.eye : 실제로 언제 쓰이나 (ML/선형대수 맥락)**
1. **초기화**: 어떤 연산이 "아무 변화도 주지 않는" 상태로 시작해야 할 때 (곱셈의 항등원 역할, 1이 곱셈에서 하는 역할과 동일)
2. **정규화(regularization)**: `A + λI` 형태로 대각선에 작은 값을 더해서 행렬을 안정화시키는 테크닉 (Ridge regression, Tikhonov regularization* 등에서 자주 등장)
3. **원-핫 인코딩(one-hot encoding)**: `np.eye(num_classes)[labels]`처럼 인덱싱해서 원-핫 벡터를 빠르게 만드는 트릭으로 자주 씀 — CS336 데이터 전처리에서도 마주칠 가능성 높음
4. **선형대수 검증**: `A @ np.linalg.inv(A)`가 실제로 `np.eye(n)`과 (근사적으로) 같은지 확인하는 용도

```python
# 모든 값이 0인 2x3 배열 생성.
zero_mat = np.zeros((2,3))
print(zero_mat, '\n')

# 모든 값이 1인 2x3 배열 생성.
one_mat = np.ones((2,3))
print(one_mat, '\n')

# 모든 값이 특정 상수인 배열 생성. 이 경우 7.
same_value_mat = np.full((2,2), 7)
print(same_value_mat, '\n')

# 대각선 값이 1이고 나머지 값이 0인 2차원 배열을 생성.
eye_mat = np.eye(3)
print(eye_mat, '\n')

# 임의의 값으로 채워진 배열 생성
random_mat = np.random.random((2,2))
print(random_mat)
```

## 1. 지도 학습(Supervised Learning)

지도 학습의 훈련 데이터는 문제지를 연상케 합니다. 지도 학습의 훈련 데이터는 정답이 무엇인지 맞춰 하는 '문제'에 해당되는 데이터와 레이블이라고 부르는 '정답'이 적혀있는 데이터로 구성되어 있습니다. 

예를 들어 스팸 메일 분류기를 위한 데이터 같은 경우에는 메일의 본문과 해당 메일이 정상 메일인지, 스팸 메일인지 적혀있는 레이블로 구성되어져 있습니다. 

예를 들어 아래와 같은 형식의 데이터가 약 20,000개 있다고 가정해보겠습니다. 이 데이터는 두 개의 열로 구성되는데, 바로 메일의 본문에 해당되는 첫번째 열과 해당 메일이 정상 메일인지 스팸 메일인지가 적혀있는 정답에 해당되는 두번째 열입니다. 그리고 이러한 데이터 배열이 총 20,000개의 행을 가집니다.

| 텍스트(메일의 내용)          | 레이블(스팸 여부) |
| -------------------- | ---------- |
| 당신에게 드리는 마지막 혜택! ... | 스팸 메일      |
| 내일 뵐 수 있을지 확인 부탁...  | 정상 메일      |
| ...                  | ...        |
| (광고) 멋있어질 수 있는...    | 스팸 메일      |

기계를 지도하는 선생님의 입장이 되어보겠습니다. 기계를 훈련시키기 위해서 데이터를 총 4개로 나눕니다. 

우선 메일의 내용이 담긴 첫번째 열을 X에 저장합니다. 
그리고 메일이 스팸인지 정상인지 정답이 적혀있는 두번째 열을 y에 저장합니다. 
이제 문제지에 해당되는 20,000개의 X와 정답지에 해당되는 20,000개의 y가 생겼습니다.

그리고 이제 이 X와 y에 대해서 일부 데이터를 또 다시 분리합니다. 
이는 문제지를 다 공부하고나서 실력을 평가하기 위해서 시험(test)용으로 일부로 일부 문제와 해당 문제의 정답지를 분리해놓는 것입니다. 
여기서는 2,000개를 분리한다고 가정하겠습니다. 이때 분리 시에는 여전히 X와 y의 맵핑 관계를 유지해야 합니다. 어떤 X(문제)에 대한 어떤 y(정답)인지 바로 찾을 수 있어야 합니다. 
이렇게 되면 학습용에 해당되는 18,000개의 X, y의 쌍과 시험용에 해당되는 2000개의 X, y의 쌍이 생깁니다 이 책에서는 이 유형의 데이터들에게 일반적으로 다음과 같은 변수명을 부여합니다.

**`<훈련 데이터>`**  
X_train : 문제지 데이터  
y_train : 문제지에 대한 정답 데이터.

**`<테스트 데이터>`**  
X_test : 시험지 데이터.  
y_test : 시험지에 대한 정답 데이터.

기계는 이제부터 X_train과 y_train에 대해서 학습을 합니다. 기계는 학습 상태에서는 정답지인 y_train을 볼 수 있기 때문에 18,000개의 문제지 X_train과 y_train을 함께 보면서 어떤 메일 내용일 때 정상 메일인지 스팸 메일인지를 열심히 규칙을 도출해나가면서 정리해나갑니다. 그리고 학습을 다 한 기계에게 y_test는 보여주지 않고, X_test에 대해서 정답을 예측하게 합니다. 그리고 기계가 예측한 답과 실제 정답인 y_test를 비교하면서 기계가 정답을 얼마나 맞췄는지를 평가합니다. 이 수치가 기계의 정확도(Accuracy)가 됩니다.

## 3. 테스트 데이터 분리하기

### 1) 사이킷 런을 이용하여 분리하기

사이킷런은 학습용 테스트와 테스트용 데이터를 쉽게 분리할 수 있게 해주는 train_test_split()를 지원합니다. => 인공지능 과제에서도 썼었음!

```python
X_train, X_test, y_train, y_test = train_test_split(X, y, test_size= 0.2, random_state=1234)
```

각 인자는 다음을 의미합니다. train_size, test_size는 둘 중 하나만 기재해도 됩니다.

>    X : 독립 변수 데이터. (배열이나 데이터프레임)  
	y : 종속 변수 데이터. 레이블 데이터.  
	test_size : 테스트용 데이터 개수를 지정한다. 
	         1보다 작은 실수를 기재할 경우, 비율을 나타낸다.  
	train_size : 학습용 데이터의 개수를 지정한다. 
	          1보다 작은 실수를 기재할 경우, 비율을 나타낸다.  
	random_state : 난수 시드

예를 들어보겠습니다. 임의로 X 데이터와 y 데이터를 생성했습니다.


train_test_split()은 기본적으로 
데이터의 순서를 섞고나서 훈련 데이터와 테스트 데이터를 분리합니다.

하지만 값을 변경하면 다른 순서로 섞인 채 분리되므로 이전과 다른 훈련 데이터와 테스트 데이터를 얻습니다. 실습을 통해서 이해해봅시다. random_state 값을 임의로 1234로 지정했습니다.

훈련 데이터의 개수와 테스트 데이터의 개수를 정해보겠습니다. num_of_train은 훈련 데이터의 개수를 의미하며, num_of_test는 테스트 데이터의 개수를 의미합니다.

```python
num_of_train = int(len(X) * 0.8) # 데이터의 전체 길이의 80%에 해당하는 길이값을 구한다.
num_of_test = int(len(X) - num_of_train) # 전체 길이에서 80%에 해당하는 길이를 뺀다.
print('훈련 데이터의 크기 :',num_of_train)
print('테스트 데이터의 크기 :',num_of_test)
```

```python

테스트 데이터의 크기 : 3
```

아직 훈련 데이터와 테스트 데이터를 나눈 것이 아니라 
이 두 개의 개수를 몇 개로 할지 정하기만 한 상태입니다. 

여기서 num_of_test를 len(X) * 0.2로 계산해서는 안 됩니다. 
데이터에 누락이 발생할 수 있습니다. 예를 들어서 전체 데이터의 개수가 4,518이라고 가정했을 때 4,518의 80%의 값은 3,614.4로 소수점을 내리면 3,614가 됩니다. 

또한 4,518의 20%의 값은 903.6으로 소수점을 내리면 903이 됩니다. 그리고 3,614 + 903 = 4517이므로 데이터 1개가 누락이 됩니다. 

그러므로 어느 한 쪽을 먼저 계산하고 그 값만큼 제외하는 방식으로 계산해야 합니다.

```python
X_test = X[num_of_train:] # 전체 데이터 중에서 20%만큼 뒤의 데이터 저장
y_test = y[num_of_train:] # 전체 데이터 중에서 20%만큼 뒤의 데이터 저장
X_train = X[:num_of_train] # 전체 데이터 중에서 80%만큼 앞의 데이터 저장
y_train = y[:num_of_train] # 전체 데이터 중에서 80%만큼 앞의 데이터 저장
```

데이터를 나눌 때는 num_of_train와 같이 하나의 변수만 사용하면 
데이터의 누락을 방지할 수 있습니다. 

앞에서 구한 데이터의 개수만큼 훈련 데이터와 테스트 데이터를 분할합니다. 그리고 테스트 데이터를 출력하여 정상적으로 분리되었는지 확인합니다.


## CSV 파일

CSV 파일은 **표(table) 데이터를 텍스트로 저장하는 가장 단순한 형식**이에요. 
데이터 다룰 때 정말 자주 만나는 거라 짚어둘 가치가 있어요.

 CSV = Comma-Separated Values
이름 그대로 **"쉼표로 구분된 값들"**이라는 뜻이에요. 표의 각 칸을 **쉼표(,)로 나누고**, 각 행을 **줄바꿈으로 나눈** 텍스트 파일입니다.

엑셀 표로 이런 데이터가 있다면:

|이름|나이|도시|
|---|---|---|
|김철수|25|서울|
|이영희|30|부산|

이걸 CSV로 저장하면 내용이 이렇게 생겼어요:

```
이름,나이,도시
김철수,25,서울
이영희,30,부산
```

- 첫 줄 = **헤더(열 이름)**
- 그 아래 = 데이터 행들
- 각 값은 **쉼표로 구분**, 각 행은 **줄바꿈으로 구분**
확장자는 `.csv`예요.

#### 핵심 특징

참고로 LLM 학습 데이터는 보통 **JSONL**(JSON Lines)이라는 형태를 써요. JSON 객체를 한 줄에 하나씩 쭉 나열하는 거예요:

#### 주의할 점 하나

값 안에 **쉼표가 들어가면** 헷갈리겠죠? 예를 들어 주소 "서울, 강남구" 같은 거. 이런 건 **따옴표로 감싸서** 구분해요:

이러면 `"서울, 강남구"`가 쉼표가 있어도 **한 칸**으로 인식됩니다. 그리고 쉼표 대신 **탭(tab)**으로 구분하면 `.tsv`(Tab-Separated Values)라고 부르는데, 원리는 같아요.

## GPU - basic 

### 출발점: 왜 구조가 다른가

CPU와 GPU는 애초에 **다른 문제를 풀려고 설계**됐어요.

- **CPU의 임무**: 다양하고 복잡한 작업을 **순서대로, 빠르게** 처리. 웹브라우저 열고, 워드 켜고, 분기문 판단하고… "다재다능한 일꾼"
- **GPU의 임무**: 원래는 그래픽(픽셀) 렌더링용. 화면의 **수백만 개 픽셀에 똑같은 계산을 동시에** 해야 했어요. "단순노동을 떼로 동시에"

이 태생적 목적 차이가 내부 구조를 완전히 갈라놨습니다.


### CPU 구조 복습 (대비용)

CPU 코어 하나는 이렇게 생겼죠 — 이미 아시겠지만 대비를 위해 짧게:

```
[CPU 코어]
├─ ALU (계산 유닛) - 작음, 코어당 몇 개뿐
├─ Control Unit (제어, 분기예측, 명령어 해석) - 매우 큼
├─ 큰 캐시(L1/L2) - 코어 전용으로 큼
└─ Out-of-order 실행, 분기예측 등 복잡한 최적화 회로
```

**핵심: 코어 1개에 "제어·예측·캐시" 같은 똑똑한 부가 회로가 잔뜩 붙어 있어요.** 그래서 코어가 비싸고 커서, 보통 **4~64개**밖에 못 박습니다. 대신 코어 하나하나가 복잡한 분기(if문), 순서 의존적인 작업을 빠르고 영리하게 처리해요.

### GPU 구조

GPU는 정반대 선택을 합니다: **코어를 단순하게 깎아서 개수를 압도적으로 늘린다.**

```
[GPU 전체]
├─ SM (Streaming Multiprocessor) #1
│    └─ 작은 코어(ALU) 수십~백 개 묶음
├─ SM #2
│    └─ 작은 코어 묶음
├─ ... (SM이 보통 수십~백 개)
└─ 공유 메모리 / VRAM (수십 GB)
```

**NVIDIA GPU 기준 용어:**
- 가장 작은 계산 단위: **CUDA 코어** (단순 ALU. 분기예측·복잡한 제어회로 거의 없음)
- 이 코어들이 묶인 단위: **SM (Streaming Multiprocessor)**
- 예를 들어 RTX 4090은 SM이 128개, CUDA 코어가 **16,384개**

CPU 코어 64개 vs GPU 코어 16,384개 — 숫자가 차원이 다르죠. 대신 **GPU 코어 하나는 CPU 코어보다 훨씬 단순하고 멍청합니다.** 분기예측도 약하고, 복잡한 순서 제어도 못해요. 할 줄 아는 건 **"곱하고 더하기" 같은 단순 산술 연산을 빠르게**입니다.


### 핵심 작동 원리: SIMT (Single Instruction, Multiple Threads)

이게 GPU 병렬화의 진짜 메커니즘이에요. **"같은 명령어 하나를, 수천 개 데이터에 동시에" 실행**하는 방식입니다.

비유로 설명하면: CPU는 **요리사 한 명이 메뉴 100개를 순서대로** 만드는 거고, GPU는 **요리사 1만 명이 "양파 썰기"라는 똑같은 동작을 동시에 1만 번** 하는 거예요. GPU 요리사들은 다른 동작을 각자 시킬 수 없어요(능력이 단순해서). 대신 **같은 동작이면 압도적으로 빠릅니다.**

신경망 계산이 정확히 이 패턴이에요. 행렬곱 Y=WXY = WX Y=WX를 떠올리면:

```
y1 = w11*x1 + w12*x2 + w13*x3 + ...
y2 = w21*x1 + w22*x2 + w23*x3 + ...
y3 = w31*x1 + w32*x2 + w33*x3 + ...
...
```

각 줄(y1, y2, y3…)을 계산하는 방식은 **완전히 똑같습니다.** 단지 곱하는 숫자만 다를 뿐이에요. 이런 "같은 연산, 다른 데이터"가 수백만 번 반복되는 게 행렬곱이고, 이게 바로 GPU가 잘하는 그 패턴이에요. CPU라면 이걸 순서대로(혹은 코어 64개로 나눠서) 처리하지만, GPU는 코어 1만 6천 개로 거의 동시에 처리해버립니다.

### 메모리 구조 — 이것도 다릅니다

CPU는 RAM에 접근하고, GPU는 **VRAM(별도의 자체 메모리)**을 씁니다.

```
[CPU] ←→ [RAM]
[GPU] ←→ [VRAM]   (GPU 카드에 붙어있는 전용 메모리)
```

앞서 본 `.to("cuda")`가 정확히 **RAM의 데이터를 VRAM으로 물리적으로 복사**하는 동작이에요. 그 후로는 GPU가 VRAM에서 직접 읽고 써요(RAM 거치면 느려서). 그래서 RAM↔VRAM 이동에는 시간이 들고(오버헤드), 한 번 옮긴 데이터는 계속 GPU에 두고 여러 연산을 시키는 게 효율적입니다.

VRAM 용량이 곧 **"얼마나 큰 모델/배치를 GPU에 올릴 수 있는가"**를 결정해요. "GPU 메모리 부족(OOM, out of memory)" 에러가 흔한 이유가 이거고, LLM 학습에서 GPU를 여러 대 쓰는 이유 중 하나도 한 GPU의 VRAM에 큰 모델이 안 들어가서예요.


### GPU 코어와 SM의 분업

조금 더 구체적으로 한 단계 더 보면:
- **SM(Streaming Multiprocessor)** 하나가 **워프(warp)** 라는 단위로 스레드를 묶어서 관리합니다(보통 32개 스레드 = 1 warp). 이 32개가 **항상 같은 명령어를 동시에** 실행해요.
- 만약 이 32개 안에서 `if` 분기가 갈리면(일부는 true, 일부는 false), GPU는 **두 경로를 순차적으로 다 실행**하고 해당 안 되는 스레드는 잠시 쉽니다. 즉 **분기가 많으면 GPU가 비효율적**이 돼요. 이게 GPU가 "분기 적고 반복 많은" 연산에만 강한 이유의 하드웨어적 근거예요.


### CPU vs GPU 한눈에 비교

|**구분**|**CPU (중앙처리장치)**|**GPU (그래픽처리장치)**|
|---|---|---|
|**코어 수**|적음 (수 ~ 수십 개)|많음 (수천 ~ 수만 개)|
|**코어 1개 성능**|강력하고 똑똑함 (분기 예측 등)|단순하고 상대적으로 약함|
|**잘하는 일**|복잡한 분기, 순서 의존적 작업|같은 연산의 대량 반복 (병렬 처리)|
|**메모리**|RAM|VRAM (별도 탑재, 보통 더 빠름)|
|**비유**|소수의 전문가가 순서대로 처리|단순 노동자 다수가 동시에 처리|
|**ML에서의 역할**|데이터 전처리, 제어 흐름 담당|행렬 곱셈, 학습 연산 담당|


### 왜 신경망이 GPU에 잘 맞나 — 마무리 연결

지금까지 본 self-attention을 떠올려보세요. QK^T 계산, softmax, V와의 가중합 
— 이거 전부 **행렬곱과 원소별 연산**이에요. 토큰이 몇 백 개든 몇 천 개든, **같은 연산을 모든 토큰·모든 차원에 동시에** 적용합니다. 분기(`if`)가 거의 없고 순수 반복 연산이라, GPU의 SIMT 구조에 완벽히 들어맞아요.

반대로 if문이 많은 일반 프로그램(웹서버 로직 같은 것)은 GPU에 올려봤자 별 이득이 없어요. **"분기 없는 대량 반복 연산이냐 아니냐"**가 CPU 갈지 GPU 갈지를 가르는 기준입니다.

---

정리: GPU는 **코어를 단순하게 깎고 개수를 폭발적으로 늘려서, 같은 명령어를 수천 개 데이터에 동시에 처리**하는 구조(SIMT)예요. 신경망의 행렬곱이 정확히 이 패턴이라 GPU가 압도적으로 빠른 거고요. `.to("cuda")`는 데이터를 VRAM으로 옮겨 이 병렬 엔진에 태우는 스위치였던 거죠.

## GPU - 멀티스레딩 체감해보기
### 멀티 GPU 동기화 — 체감하고 싶다면

좋은 본능이에요. "수가 많아서 가벼워 보여도 합치면 비용 크다"는 직관이 정확히 맞고, 이건 글로 백날 읽는 것보다 **숫자로 한 번 재보는 게** 훨씬 와닿습니다. 다만 현실적 제약부터 먼저 말씀드릴게요.

#### 현실적 제약: GPU 2개 이상이 물리적으로 필요

CPU의 멀티스레딩 공유자원 예제는 노트북 한 대로 됩니다(코어가 이미 여러 개니까). 하지만 멀티 GPU 통신을 체감하려면 **GPU가 최소 2개** 있어야 해요. 개인 PC에 GPU 2장 박혀 있는 경우는 드물죠. 그래서 선택지가 좁아져요.

**선택지 A — 클라우드로 GPU 여러 개 잠깐 빌리기**  
가장 현실적이에요. Google Colab Pro(가끔 multi-GPU 제공), 또는 Lambda Labs/RunPod/Vast.ai 같은 곳에서 **A100×2 같은 인스턴스를 시간 단위로** 빌릴 수 있어요. 한두 시간 빌려서 실험 한 번 돌리는 거면 비용이 크지 않습니다(몇천 원~몇만 원 수준).

**선택지 B — CS336 실습/클러스터**  
앞서 얘기 나눈 "cluster를 provide" 받는 그 자원을 이 목적으로 정확히 써볼 수 있어요. CS336 후반 과제가 분산학습(distributed training)을 다룰 가능성이 높습니다. 강의가 이걸 위해 만들어진 인프라니, 가장 자연스러운 경로예요.

#### 체감 실험 설계 (GPU 빌렸을 때)

CPU 공유자원 예제처럼 **"먼저 동기화 없이 망가뜨려보고, 그 다음 동기화로 고쳐보는"** 순서를 추천해요.

**1단계 — 통신 비용을 숫자로 직접 재기 (가장 쉬움, 추천 시작점)**

PyTorch 분산통신 기본 함수 `all_reduce`로, GPU 여러 개가 텐서를 주고받는 시간을 직접 측정하는 거예요.

python

```python
import torch
import torch.distributed as dist
import time

dist.init_process_group("nccl")
rank = dist.get_rank()
tensor = torch.randn(10_000_000, device=f"cuda:{rank}")

torch.cuda.synchronize()
start = time.time()
dist.all_reduce(tensor)   # GPU들끼리 텐서를 합산 동기화
torch.cuda.synchronize()
print(f"통신 시간: {time.time()-start:.4f}초")
```

이걸 텐서 크기를 키워가며(100만 → 1억 원소) 돌려보면, **"무게가 별로 안 느껴져도 합치면 비용 크다"는 직관이 그래프로 바로 보여요.** GPU 내부 계산은 ms 단위인데, 통신은 그보다 훨씬 크게 튀는 지점을 직접 관찰하게 됩니다.

**2단계 — 동기화 없이 망가뜨려보기 (체감 핵심)**

데이터 병렬 학습을 흉내내되, **일부러 `all_reduce`를 빼고** 각 GPU가 자기 gradient로만 따로 업데이트하게 해보세요. 그럼 두 GPU의 모델 가중치가 점점 **서로 달라지는** 걸 관찰할 수 있어요(`print(model.state_dict())`로 GPU별 비교). 그 다음 `all_reduce`를 다시 넣어서 가중치가 일치 상태를 유지하는 걸 비교하면, "왜 동기화가 필요한지"가 눈에 보입니다. CPU 전역변수 레이스 컨디션 예제랑 정확히 같은 교훈을 GPU 버전으로 재현하는 셈이에요.


**3단계 — 실전 라이브러리로 가속 체감**

PyTorch의 `DistributedDataParallel`(DDP)을 켜고 끄고 GPU 2개로 같은 학습을 돌려서, "동기화 오버헤드를 자동 관리해주는 라이브러리가 있고 없고"의 차이를 시간으로 비교.

=> 이론을 배운 다음에 해보기



#### CPU 비유와의 차이점 정리 (이해 갭 메우기)

질문하신 "GPU끼리 어떻게 저글링하는지 감이 덜 온다"는 부분, 핵심은 **공유 메모리가 없다**는 거예요.

- **CPU 멀티스레딩**: 스레드들이 **같은 RAM**을 보고 있어서, 전역변수 하나를 여러 스레드가 직접 읽고 씁니다. 그래서 lock·mutex로 "동시에 건드리지 마라"를 막는 게 핵심 이슈.
- **멀티 GPU**: 각 GPU가 **자기만의 VRAM**을 갖고 있어서(물리적으로 분리), 애초에 서로의 메모리를 못 봅니다. 그래서 문제는 "동시에 건드리지 마라"가 아니라 **"내 결과를 남한테 보내고, 남의 결과를 받아서 합쳐라"**는 명시적 **메시지 전달(message passing)**이에요. 이게 `all_reduce`가 하는 일입니다 — 각 GPU의 gradient를 모아서 평균 내고 다시 모든 GPU에 뿌리는 것.

비유하면 CPU는 **한 사무실에서 같은 화이트보드를 여럿이 보며 일하는 것**(누가 언제 쓰냐가 문제), GPU는 **각자 다른 사무실에서 일하다가, 정기적으로 결과를 우편으로 주고받아 맞추는 것**(우편 배달 자체가 시간 드는 게 문제)에 가까워요. 그래서 lock 개념보다는 **네트워크 통신 지연**에 더 가까운 직관으로 접근하셔야 맞습니다.

## GPU 간의 통신
 
 어떻게 하는가? bus? LAN? WAN?

좋은 지점을 짚으셨어요. 결론부터: **인터넷(공인망)을 경유하진 않지만, 본질적으로는 네트워크 통신이 맞습니다.** 다만 "네트워크"라는 단어가 주는 이미지보다 훨씬 더 가깝고 빠른 통신이에요. 단계별로 풀게요.

### 물리적으로 GPU끼리 어떻게 연결되어 있나 — 3단계 거리
같은 서버 안인지, 다른 서버인지에 따라 통신 경로가 완전히 달라집니다. 
**가까운 순서대로** 보면:

#### ① 같은 서버, GPU끼리 직접 연결 — NVLink

가장 빠른 경로예요. NVIDIA GPU 여러 장이 한 서버에 꽂혀 있을 때, **GPU와 GPU를 직접 잇는 전용 케이블/버스**가 있어요. 이게 NVLink입니다.

```
[GPU 0] ═══ NVLin k═══ [GPU 1]
   (PCIe 거치지 않고 GPU끼리 직통)
```

비유하면 두 사무실 사이에 **전용 직통 파이프**를 깔아둔 거예요. 우편(인터넷)도 아니고, 심지어 같은 건물 복도(PCIe)도 안 거치고, **둘만 잇는 초고속 전용선**입니다. 그래서 같은 서버 안 GPU끼리는 이게 있으면 제일 빠르고, 없으면(보급형 GPU) 한 단계 아래로 내려갑니다.

#### ② 같은 서버, NVLink 없을 때 — PCIe

NVLink가 없는 GPU들은 **메인보드의 PCIe 버스**를 통해 통신해요. 이건 GPU가 CPU·RAM이랑 연결될 때 쓰는 그 슬롯, 그 길을 GPU끼리 통신할 때도 같이 씁니다(보통 CPU를 한 번 거쳐서). NVLink보다 느려요.


#### ③ 서버가 다를 때 — 진짜 네트워크(랜선/광케이블)

GPU 8장짜리 서버가 여러 대 있어서(예: 32개 GPU로 학습) 서버를 넘나들어야 하면, 여기서부터가 질문하신 "랜으로?"에 해당해요. **InfiniBand**라는 초고속 전용 네트워크 케이블(일반 이더넷보다 훨씬 빠름)로 서버들을 직결합니다. 이건 일반 인터넷망이 아니라 **데이터센터 내부에만 깔린 전용 고속망**이에요. 일반 가정용 인터넷이 보통 수백 Mbps~수 Gbps라면, InfiniBand는 **수백 Gbps**급입니다.

```
거리/속도:  NVLink(가장빠름) > PCIe > InfiniBand(서버간) > 일반 인터넷(가장느림, 안씀)
```

핵심은 **인터넷(공인망)은 절대 안 거칩니다.** 보안 문제도 있고 무엇보다 너무 느려서 학습 자체가 불가능해요. 전부 데이터센터 **내부의 전용선**입니다.


### 소프트웨어 레벨에서는 어떻게 "소통"하나

물리선은 위와 같고, 그 위에서 실제로 데이터를 주고받는 **프로토콜/소프트웨어**가 있어요. 앞서 언급한 **NCCL**(NVIDIA Collective Communication Library)이 이 역할이에요.

`all_reduce`를 부르면 내부적으로 이런 일이 일어나요:

```
GPU 0: gradient = [1.0, 2.0, 3.0]
GPU 1: gradient = [4.0, 5.0, 6.0]
GPU 2: gradient = [7.0, 8.0, 9.0]
GPU 3: gradient = [2.0, 1.0, 0.0]

   ↓  all_reduce (NCCL이 알아서 GPU들을 링 모양 등으로 연결해 전송)

모든 GPU: gradient = [3.5, 4.0, 4.5]   ← 평균낸 값이 전부에게 똑같이 도착
```

NCCL은 GPU들을 **링(ring)** 형태로 연결해서, 옆 GPU에게만 데이터를 넘기는 식으로 효율적으로 전체에 퍼뜨리는 알고리즘(ring all-reduce)을 씁니다. 모든 GPU가 모든 GPU에 직접 쏘는 게 아니라, 옆으로만 릴레이하면서 결국 전체가 합쳐지는 거예요. 이게 통신량을 최소화하는 핵심 트릭입니다.

### CPU 네트워크 프로그래밍과 비교하면


|              | CPU 멀티스레딩           | 멀티 GPU                                                    |
| ------------ | ------------------- | --------------------------------------------------------- |
| 메모리          | 공유 RAM (포인터로 직접 접근) | 각자 독립 VRAM                                                |
| 통신 방식        | 메모리 직접 read/write   | 메시지 전송 (NVLink/PCIe/InfiniBand)                           |
| 동기화 도구       | mutex, lock         | all_reduce, NCCL                                          |
| 가장 가까운 OS 비유 | 스레드 (공유 메모리)        | **프로세스 간 통신(IPC)**, 혹은 더 정확히는 **분산 시스템(여러 컴퓨터가 메시지로 협업)** |

즉 멀티 GPU는 "멀티스레딩"보다는 **"분산 컴퓨팅(distributed computing)"** 에 훨씬 가까운 모델이에요. 같은 컴퓨터 안 스레드끼리 메모리 공유하는 게 아니라, **독립된 노드들이 메시지를 주고받아 협업**한다는 점에서요. 그래서 분산 시스템 수업에서 배우는 "노드 간 동기화, latency, bandwidth" 같은 개념이 멀티 GPU에 그대로 적용됩니다.


### 정리

- 일반 인터넷은 절대 안 거침. 전부 **데이터센터 내부 전용 초고속 연결**(NVLink → PCIe → InfiniBand 순으로 빠름).
- 소프트웨어로는 **NCCL** 같은 라이브러리가 "누구한테 언제 뭘 보낼지"를 자동 관리.
- 개념적으로는 **멀티스레딩(공유메모리)보다 분산시스템(메시지전달)에 가까움.**

흥미로우시면, 앞서 말씀드린 `all_reduce` 실습을 GPU 2개짜리 클라우드 인스턴스에서 돌려보면 이 NCCL 통신 시간이 실제로 찍혀요. NVLink 있는 인스턴스 vs 없는 인스턴스를 비교해보면 "거리에 따라 속도 차이"도 직접 체감되실 거예요.



## 추가 (CUDA, GPU 종류별 구조 등)
#### CUDA = 로우레벨/HW 재설계 기전인가

방향은 맞는데 **레이어를 하나 더 나눠야** 정확해요.

```
PyTorch (Python)
   ↓
PyTorch C++ 엔진 (ATen)
   ↓
CUDA (C/C++ 확장 언어 + 런타임 API)   ← 사람이 이 레벨에서 코드를 짤 수 있음
   ↓
PTX (NVIDIA의 가상 어셈블리, GPU 종류 무관한 중간언어)
   ↓
SASS (실제 그 GPU 모델 전용 진짜 어셈블리)
   ↓
하드웨어
```

CUDA 자체는 **어셈블리가 아니라 C++의 확장 언어**예요. `__global__ void kernel(...)` 같은 문법으로 "이 함수를 GPU 코어 수천 개에 나눠 돌려라"라고 짭니다. 사람이 직접 짜는 가장 낮은 레벨이 보통 여기까지고, 그 아래(PTX→SASS, 진짜 어셈블리)는 NVIDIA 컴파일러(nvcc)가 자동 생성해요. 그러니 "정교히 재설계된 로우레벨 기전"이라는 직관은 맞지만, **어셈블리 자체는 아니고 그 한 단계 위**입니다.

#### GPU 종류마다 구조가 다른가
**네, 벤더마다 용어와 세부 구현이 달라요.**

- NVIDIA: CUDA 코어, SM, CUDA(소프트웨어 생태계)
- AMD: Stream Processor, Compute Unit(CU), ROCm(소프트웨어 생태계)
- Intel: Xe 코어 등

이름은 다르지만 **개념(많은 단순 코어 + SIMT 병렬)은 거의 동일**해요. 다만 ML 생태계는 CUDA가 압도적 표준이라(라이브러리·최적화가 다 CUDA 기준), 실무에서는 사실상 NVIDIA GPU만 쓴다고 봐도 무방합니다.


#### SM = ALU만 묶은 것, 제어장치 부담 없앤 것인가

**완전히 맞진 않아요, 정정할게요.** SM도 제어 로직이 있긴 해요. 다만 **CPU처럼 코어마다 따로 있는 게 아니라, "워프(32개 스레드)"를 묶어서 하나의 제어 로직이 공동으로 관리**합니다.

비유: CPU는 직원 1명에 매니저 1명(코어마다 독립 제어). GPU는 매니저 1명이 32명에게 **"전부 똑같은 동작 하나를 동시에 해"**라고 명령(SM 하나가 워프 단위로 묶어 제어). 그래서 **제어회로 자체가 없어진 게 아니라, 1:1에서 1:32(혹은 그 이상)로 공유 비율이 확 줄어든 것**이에요. 그 절약된 트랜지스터 예산을 ALU 개수 늘리는 데 쓴 거고요.


#### VRAM = 계산 전용 RAM인가

맞아요, 그리고 이유 하나 더 있어요. VRAM(보통 GDDR/HBM)은 일반 RAM보다 **대역폭(bandwidth)이 훨씬 큽니다.** GPU는 수천 코어가 동시에 데이터를 요구하니까, "한 번에 얼마나 많은 데이터를 퍼부어줄 수 있느냐"가 중요해서, 일반 RAM보다 대역폭에 특화된 메모리를 씁니다. "계산 전용"이라는 직관 + "퍼붓기 전용 대역폭"까지 더하면 정확해요.


#### 보호 vs 공유 — 이 부분은 정정이 필요해요

이 직관은 **틀렸어요.** VRAM이 "보호를 포기하고 공유를 택한" 게 아닙니다. 현대 GPU도 **가상 메모리, 프로세스 간 격리(isolation)를 갖고 있어요.** 여러 프로세스가 같은 GPU를 동시에 쓸 때 서로 메모리를 침범 못 하게 막는 보호 장치가 존재합니다(다만 CPU만큼 정교하진 않고, 역사적으로 더 늦게 추가됨).

진짜 차이는 **"보호 vs 공유"가 아니라 "제어 흐름의 단위"** 예요. RAM/CPU는 **스레드 하나하나가 독립적으로 다른 코드를 실행**할 수 있어요(그래서 보호·스케줄링이 복잡). VRAM/GPU는 **수천 스레드가 거의 항상 같은 코드를 동시에** 돌리니, 보호가 필요 없어서가 아니라 **애초에 서로 다른 일을 안 시켜서** 충돌 우려 자체가 구조적으로 적은 거예요.


#### GPU엔 컨텍스트 스위치가 없는가

**부분적으로만 맞아요.** OS가 프로세스를 바꿔치기하는 그런 무거운 컨텍스트 스위치는 GPU에 없습니다. 대신 GPU는 **워프 스위칭**이라는 자체 메커니즘이 있어요: 한 워프가 메모리 읽기를 기다리며 멈추면(stall), SM이 **그 자리에서 즉시 다른 준비된 워프로 갈아탑니다.** 이게 OS 컨텍스트 스위치처럼 무겁지 않고 거의 비용이 없는(zero-overhead) 이유는, **모든 워프의 레지스터가 이미 칩 위에 동시에 다 올라와 있기 때문**이에요(저장/복원할 필요가 없음). 그래서 "컨텍스트 스위치가 없다"가 아니라 **"공짜에 가까운 컨텍스트 스위치가 있다"**가 더 정확해요.



#### to(cuda) = fork()처럼 메모리 복사하는 건가

**비슷한 결은 맞지만 이유는 달라요.** fork()는 "보안/격리를 위해" 메모리를 복사하는 거지만, `.to("cuda")`는 **단순히 물리적으로 다른 메모리 칩(RAM↔VRAM)이라서** 복사가 필요한 거예요. GPU와 CPU는 **물리적으로 분리된 메모리 공간**(PCIe 버스로 연결됨)이라, RAM에 있는 데이터를 GPU가 쓰려면 VRAM으로 실제 전송해야만 합니다. 보안 목적이 아니라 **"애초에 다른 칩이라 거기 있는 데이터를 못 읽으니까"**가 이유예요. 그래서 fork보다는 **"네트워크 너머의 다른 서버로 파일 전송하는 것"**에 더 가까운 비유예요.


#### 멀티 GPU = 멀티코어처럼 쓰는 건가, 문제는 없나

맞는 방향인데, **문제가 꽤 있어요.** 이게 LLM 학습이 어려운 이유 중 하나입니다.

- GPU끼리는 **CPU 코어들보다 통신이 훨씬 느려요.** 같은 메인보드 안 GPU끼리도(NVLink 같은 전용 연결 없으면) 데이터 주고받는 데 시간이 꽤 듭니다.
- 모델을 여러 GPU에 쪼개면(파라미터를 나눠 올림), 학습 중 계속 **GPU 간 데이터 동기화**가 필요해요(예: gradient 합산). 이 통신이 병목이 되기 쉬워요.
- 그래서 NCCL 같은 전용 통신 라이브러리, 그리고 "어떻게 모델을 쪼개야 통신을 최소화하나"(데이터 병렬·모델 병렬·파이프라인 병렬 등) 자체가 큰 연구 주제예요.

즉 "멀티코어처럼 그냥 나눠 쓴다"보다 훨씬 까다롭고, **통신 오버헤드가 멀티GPU 학습의 진짜 난제**입니다.
=> 하긴 멀티프로세싱? 멀티스레딩. 을 대규모로 하는데 sync 문제가 없을리가 없다. CPU만큼 큰 비용이라고 해야 할까 .. weight 가 크게 느껴지진 않을거같아도 얘네가 수가 많다보니까 여러개를 동시에 진행하면 GPU끼리의 데이터동기화가 있어야 할 것 같긴 하다. 이건 꼭 실습으로 경험해보고 싶음. 방법 :



#### if 분기 — assem 수준의 if분기로 hazard 생기는상황인가?

맞습니다, 워프 다이버전스(warp divergence).
정확히 그 얘기예요. 다만 "어셈블리 hazard"는 아니고, 별개의 GPU 고유 현상이에요. 정리하면:

워프(32스레드)는 **항상 같은 명령어를 동시에** 실행해야 하는데, 만약 코드에:

```
if (x > 0) { A 실행 }
else       { B 실행 }
```

이 있고, 32개 스레드 중 20개는 조건이 참, 12개는 거짓이라면 
— GPU는 **A를 20개로 실행하는 동안 나머지 12개는 그냥 놀고(masked out), 
끝나면 B를 12개로 실행하는 동안 20개가 놉니다.** 

즉 **A와 B를 순차적으로 둘 다 돌리면서, 해당 안 되는 스레드는 결과를 버려요.** 그래서 분기가 갈리면 실질적으로 **(A 걸리는 시간 + B 걸리는 시간)이 그대로 더해져서 손해**가 나는 거예요. 이게 CPU의 파이프라인 해저드(분기예측 실패로 인한 정정)와는 다른, **GPU 고유의 비효율 메커니즘**입니다.


#### Fortran — 아직 씁니다, 의외로

고전언어 맞는데 **죽지 않고 과학계산 분야에 단단히 박혀 있어요.** 이유는 두 가지예요.

**① 레거시 + 검증된 신뢰성**: 수십 년간 쓰인 기상예측·물리시뮬레이션·선형대수 코드가 Fortran으로 짜여 있고, 이미 극한까지 최적화·검증된 코드를 새로 짤 이유가 없어요. NumPy/SciPy의 행렬연산 핵심부(LAPACK, BLAS)도 지금도 Fortran입니다. 결국 우리가 방금 본 NumPy 행렬곱도, 한 꺼풀 더 들어가면 Fortran 코드를 호출하고 있을 가능성이 커요.

**② Fortran은 배열 연산에 특화된 문법**이라, 일부 수치계산에서는 C보다도 컴파일러가 최적화하기 쉬워요. 그래서 "느려서 못 버린" 게 아니라 "여전히 이 영역에선 잘함"에 가깝습니다.



#### 어셈블리 코더, 요즘 없나 — 거의 맞습니다

말씀하신 인과관계(컴파일러 이론·최적화 발전 → 사람이 어셈블리 직접 짤 이유가 없어짐)가 정확해요. 다만 **완전히 0은 아니고, 남은 영역이 셋 정도** 있어요.

- **컴파일러 자체를 만드는 사람들**: 결국 누군가는 "이 고수준 코드를 어떤 어셈블리로 변환할지" 그 규칙을 짭니다.
- **극한 최적화가 필요한 좁은 영역**: 게임엔진의 가장 뜨거운 루프, 임베디드(자원 극히 제한된 마이크로컨트롤러), 보안 분야(리버스 엔지니어링, 익스플로잇 분석)는 지금도 어셈블리를 직접 들여다봅니다.
- **컴파일러가 못 짜내는 특수 명령어**: GPU의 SASS, CPU의 SIMD intrinsic 같은 건 컴파일러 자동생성 + 사람이 미세조정을 같이 합니다.

근데 일반적인 ML/SW 개발 범위에선 말씀대로 **사실상 안 합니다.** 정확한 그림이에요.




## Ridge regression, Tikhonov regularization

둘은 사실 **거의 같은 개념**이고, 부르는 이름만 분야마다 다른 겁니다.

**공통 아이디어부터**
일반적인 선형회귀(linear regression)는 다음을 최소화합니다:

```
Loss = ||Xw - y||²   (예측값과 실제값의 오차 제곱합)
```

근데 이것만 최소화하면 **과적합(overfitting)** 문제가 생깁니다. 
특히 특성(feature)이 많거나 데이터가 적을 때, w가 너무 크고 들쭉날쭉한 값을
가지면서 학습 데이터엔 딱 맞지만 새 데이터엔 예측을 못하는 모델이 나오기 쉬워요.

여기에 **"w가 너무 커지지 않도록 벌점(penalty)을 추가"**하는 게 
핵심 아이디어입니다:

```
Loss = ||Xw - y||² + λ||w||²
```

여기서 λ(람다)는 벌점 강도를 조절하는 하이퍼파라미터, ||w||²는 w 벡터 크기의 제곱(L2 norm)입니다.


### **Ridge regression** (릿지 회귀)

- 위 식을 그대로 부르는 **통계학/머신러닝 쪽 용어**
- 이 문제를 풀면 closed-form 해가 다음과 같이 나옵니다:

```
w = (XᵀX + λI)⁻¹ Xᵀy
```

여기서 `λI`(람다 곱하기 단위행렬)가 등장하죠 — 바로 앞서 얘기한 `np.eye`가 여기 쓰이는 겁니다. 
XᵀX에 λI를 더해주는 이유는, XᵀX가 역행렬이 존재하지 않거나(특이행렬) 수치적으로 불안정한 경우가 많은데, 대각선에 작은 양수를 더해주면 행렬이 항상 역행렬을 가지도록(수치적으로 안정화) 만들어주기 때문입니다.



**Tikhonov regularization**

- **수학/공학(특히 역문제, inverse problem) 쪽 용어**
- 안드레이 티호노프(Andrey Tikhonov)라는 소련 수학자가 1940~60년대에 "잘 정의되지 않은(ill-posed) 문제"를 풀 때 제안한 일반적인 정규화 기법
- 수식적으로 완전히 동일: `||Xw - y||² + λ||Γw||²` (Γ는 보통 단위행렬을 씀 → Ridge와 동일해짐)

**정리하면**

|Ridge Regression|Tikhonov Regularization|
|---|---|---|
|분야|통계학, ML|수학, 응용수학, 신호처리|
|시대|1970년 (Hoerl & Kennard)|1940~60년대 (Tikhonov)|
|수식|사실상 동일|사실상 동일 (더 일반적 형태 포함)|

같은 수학적 아이디어가 서로 다른 커뮤니티에서 독립적으로 발전하면서 이름이 두 개가 붙은 케이스입니다. 통계/ML 논문에서는 "Ridge", 수치해석/역문제 논문에서는 "Tikhonov"라고 부르는 경향이 있는데, 최근엔 ML 쪽에서도 "L2 regularization"이라는 더 일반적인 이름으로 부르는 경우가 많습니다 (딥러닝에서 weight decay라고 부르는 것도 본질적으로 같은 원리).
